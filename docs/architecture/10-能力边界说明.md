## 1. 能力概述

### 1.1 我能做什么

```
✅ 架构设计:
   - 设计系统架构
   - 模块划分
   - 技术选型
   - 设计模式应用

✅ 代码实现:
   - 编写业务代码
   - 实现数据层
   - 实现 UI 组件
   - 编写测试代码

✅ 问题诊断:
   - 分析错误日志
   - 定位 Bug 原因
   - 提供修复方案
   - 性能分析

✅ 文档编写:
   - 技术文档
   - API 文档
   - 使用指南
   - 注释说明

✅ 代码审查:
   - 代码质量检查
   - 最佳实践建议
   - 重构建议
   - 安全检查
```

### 1.2 我不能做什么

```
❌ 无法直接访问:
   - 本地文件系统（需要通过工具）
   - 数据库（需要通过代码）
   - 网络 API（需要通过代码）
   - 第三方服务（需要通过 SDK）

❌ 无法执行:
   - 直接运行代码
   - 构建项目
   - 部署应用
   - 执行系统命令（通过 Bash 工具）

❌ 无法保证:
   - 代码 100% 正确（需要人工审核）
   - 性能最优（需要测试验证）
   - 安全无漏洞（需要专业审计）
```

## 5. 问题诊断能力

### 5.1 能诊断的问题类型

```
✅ 编译错误:
   - 语法错误
   - 类型错误
   - 依赖错误

✅ 运行时错误:
   - 空指针异常
   - 状态错误
   - 网络错误

✅ 性能问题:
   - 卡顿分析
   - 内存泄漏
   - 过度渲染

✅ 架构问题:
   - 循环依赖
   - 模块耦合
   - 设计缺陷
```

### 5.2 诊断流程

```
1. 分析错误信息
2. 定位问题代码
3. 分析问题原因
4. 提供解决方案
5. 预防类似问题
```

### 5.3 诊断局限

```
⚠️ 需要:
   - 完整的错误日志
   - 相关的代码片段
   - 复现步骤
   - 环境信息

⚠️ 无法:
   - 100% 确定问题原因
   - 保证解决方案有效
   - 替代实际调试
```

## 7. 局限性

### 7.1 技术局限

```
⚠️ 无法:
   - 直接运行和测试代码
   - 访问实时的运行状态
   - 与外部服务直接交互
   - 感知真实用户反馈

⚠️ 依赖:
   - 提供的信息准确性
   - 代码上下文完整性
   - 问题描述清晰度
```

### 7.2 知识局限

```
⚠️ 知识截止:
   - 训练数据有时间限制
   - 最新的技术可能不了解
   - 某些小众技术可能不熟悉

⚠️ 经验限制:
   - 缺乏实际项目经验
   - 无法感知团队动态
   - 无法理解业务复杂性
```

### 7.3 判断局限

```
⚠️ 可能:
   - 理解错误的需求
   - 提供不够优化的方案
   - 忽略某些边界情况
   - 建议不合适的技术

⚠️ 需要:
   - 人工审核和验证
   - 实际测试和调整
   - 团队讨论和决策
```

## 8. 最佳实践

### 8.1 如何最好地使用我

```
✅ 最佳实践:
   1. 提供清晰完整的需求
   2. 给出足够的上下文
   3. 提供相关的代码示例
   4. 明确期望的输出格式
   5. 逐步验证和反馈

✅ 沟通方式:
   - 具体而非抽象
   - 分步骤而非一次性
   - 带示例而非纯描述
   - 有约束而非开放式
```

### 8.2 工作流程建议

```
1. 需求讨论
   - 明确业务需求
   - 讨论技术方案
   - 确定技术栈

2. 架构设计
   - 设计系统架构
   - 划分模块边界
   - 定义接口规范

3. 代码实现
   - 生成基础代码
   - 实现业务逻辑
   - 编写测试代码

4. 代码审查
   - 检查代码质量
   - 优化性能
   - 修复问题

5. 文档维护
   - 更新技术文档
   - 完善 API 文档
   - 记录变更日志
```

### 8.3 质量保证

```
✅ 必须人工审核:
   - 架构设计方案
   - 核心业务逻辑
   - 安全相关代码
   - 性能关键路径

✅ 必须测试验证:
   - 生成的代码
   - 修复的 Bug
   - 优化的方案
   - 集成的功能

✅ 持续改进:
   - 收集反馈
   - 调整方案
   - 优化流程
   - 积累经验
```

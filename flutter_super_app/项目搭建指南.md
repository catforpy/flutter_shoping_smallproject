## 🏗️ 创建流程说明

### 第一步：创建空的分类文件夹

```bash
# 这些是空的文件夹，用来分类组织代码
mkdir -p packages/core          # 存放基础层包
mkdir -p packages/services      # 存放服务层包
mkdir -p packages/modules       # 存放业务模块包
mkdir -p packages/apps          # 存放应用
```

**为什么要这样做？**
```
就像整理衣柜：
1. 先准备不同的抽屉（创建文件夹）
   ├── 上衣抽屉
   ├── 裤子抽屉
   └── 袜子抽屉

2. 然后往抽屉里放衣服（创建 Package）
   ├── 上衣抽屉/衬衫
   ├── 上衣抽屉/T恤
   └── 裤子抽屉/牛仔裤

这样分类清楚，方便找！
```

### 第二步：用 Flutter 命令创建具体的 Package

```bash
# 在 core 文件夹里创建基础包
cd packages/core
flutter create --template=package core_base    # 创建基础类型包
flutter create --template=package core_utils   # 创建工具函数包
flutter create --template=package core_network # 创建网络包
```

**flutter create 命令会自动生成：**
```
core_base/
├── lib/                    # 代码文件夹
│   └── core_base.dart      # 主文件
├── test/                   # 测试文件夹
├── pubspec.yaml            # 配置文件（类似 package.json）
└── README.md               # 说明文件
```

### 第三步：配置依赖关系

**在 pubspec.yaml 文件里添加依赖：**

```yaml
# core_network/pubspec.yaml
name: core_network
description: 网络请求包

dependencies:
  flutter:
    sdk: flutter

  # 依赖基础包（本地路径）
  core_base:
    path: ../core_base
  core_utils:
    path: ../core_utils

  # 依赖第三方库
  dio: ^5.4.0                # 网络请求库
```

**依赖关系说明：**
```
core_network（网络包）依赖：
  └── core_utils（工具包）
      └── core_base（基础包）

就像：
盖房子依赖：
  └── 砖头
      └── 泥土

必须先有底层，才能有上层！
```

## 🎯 各层 Package 的作用

### 📦 基础设施层（core/）

| Package | 中文名 | 作用 | 依赖 |
|---------|--------|------|------|
| **core_base** | 基础类型 | 定义最基础的类型，比如：Result（结果）、Either（两者之一） | 无 |
| **core_utils** | 工具函数 | 提供各种工具方法，比如：字符串转日期、手机号验证 | core_base |
| **core_constants** | 常量定义 | 定义所有常量，比如：API地址、默认配置 | 无 |
| **core_exceptions** | 异常处理 | 定义所有错误类型，比如：网络错误、认证错误 | core_base |
| **core_logging** | 日志系统 | 记录程序运行信息，方便调试 | 无 |

### 🔧 核心能力层（core/ 续）

| Package | 中文名 | 作用 | 依赖 |
|---------|--------|------|------|
| **core_network** | 网络请求 | 处理所有HTTP请求，访问API接口 | core_utils, core_base |
| **core_database** | 数据库 | 本地数据存储，SQLite数据库 | core_utils |
| **core_cache** | 缓存系统 | 临时存储数据，提升性能 | core_database |
| **core_state** | 状态管理 | 管理应用的数据状态（使用Riverpod） | core_utils |
| **core_router** | 路由管理 | 管理页面跳转 | 无 |

### 🏪 业务服务层（services/）

| Package | 中文名 | 作用 | 依赖 |
|---------|--------|------|------|
| **service_user** | 用户服务 | 用户注册、登录、个人信息 | core_network, core_database |
| **service_order** | 订单服务 | 订单创建、查询、更新 | core_network, core_database |
| **service_payment** | 支付服务 | 微信支付、支付宝支付 | core_network |

### 🎨 业务模块层（modules/）

| Package | 中文名 | 作用 | 依赖 |
|---------|--------|------|------|
| **module_shop** | 电商模块 | 商品浏览、购物车、下单 | service_user, service_order |
| **module_live** | 直播模块 | 直播间、推流、观看 | service_user, service_live |
| **module_im** | 即时通讯 | 聊天、群聊、消息 | service_user, service_im |

### 📱 应用层（apps/）

| Package | 中文名 | 作用 | 依赖 |
|---------|--------|------|------|
| **app_main** | 主应用 | 综合应用，包含所有功能 | 所有模块 |
| **app_shop** | 电商应用 | 独立的购物应用 | module_shop |

```
应用层
    ↓ 依赖
业务模块层
    ↓ 依赖
业务服务层
    ↓ 依赖

```

```
必须从下往上建！
```

## ❓ 常见问题

### Q1: 为什么要分这么多层？
**A**: 就像商场分层，便于管理。如果所有东西都放一起，40多个包会非常混乱。

### Q2: 可以合并一些层吗？
**A**: 可以，但不推荐。你的项目很复杂，分层清晰能避免很多问题。

### Q3: 必须用英文名称吗？
**A**: 是的，这是 Dart/Flutter 的规范。中文文件名可能会有问题。

### Q4: 忘记某个 Package 是干什么的怎么办？
**A**: 查看本文档的"各层 Package 的作用"部分，或者看 Package 里的 README.md 文件。
